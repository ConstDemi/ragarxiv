# RAGarxiv — Дорожная карта

Научный RAG-ассистент для навигации по NLP-статьям arXiv. Область:
- **Q1** — «существует ли статья о X» (наличие работы / датасета / бенчмарка);
- **Q2** — «какие работы помогают с проблемой X и как» (метод + результат).

> ## ⚠️ Это живой документ, а не догма
>
> Карта — это **текущая гипотеза о порядке работ**, выведенная из сегодняшних данных,
> а не контракт и не непреложная истина. Её **можно и нужно пересматривать**: переставлять
> фазы, выкидывать пункты, добавлять новые, менять приоритеты. Если данные противоречат
> предположению фазы — побеждают данные, а не карта. Конкретные триггеры для пересмотра —
> в разделе [«Когда и как пересматривать»](#когда-и-как-пересматривать-эту-карту) в конце.
> Не считай ни одну стрелку здесь обязательной.

---

## Принцип

**Каждая фаза открывается доказательством необходимости, и функцию не строим, пока оценка не сможет измерить её выигрыш.**

Следствия:
- Бьём туда, где **данные** показывают потерю, а не туда, где «принято» или «модно».
- Эксперименты — прицельные A/B по **одному изменению**; разница засчитывается, только если превышает шум (~±0.05 при n=50).
- **Версия оценки** (набор вопросов / модель судьи / что видит судья) и версия системы — разные вещи; суффикс `vN` в именах прогонов MLflow отмечает версию оценки, не систему. При смене версии оценки прогоны между собой несравнимы.

## Фаза 0 — Генерация (закрыта)

**Цель:** дожать faithfulness и answer_relevancy на **фиксированных** корпусе/наборе/судье.

**Рычаги, по одному за итерацию:**
1. системный промпт с упором на опору на контекст (граница «данные/инструкции» + язык ответа + калибровка отказа) — *пробовали → ухудшение, откат*;
2. если устоит подмена инструкций из контекста (#42) → усиление шаблона контекста (явные разделители);
3. если устоят выдумки (#15) → принудительное цитирование источников;
4. язык ответа → директива в конце сообщения пользователя, если промпт не помог.

**Критерий выхода:** метрики вышли на плато в пределах шума (~±0.05) две итерации подряд; остаток ошибок упирается в поиск (промптом не чинится).

**Итог (критерий достигнут):** `prompt-grounded-v2` (4 правила блоком) → ухудшение (faithfulness 0.83→0.70, чрезмерные отказы + срыв языка); `prompt-lang-v2` (правило о языке) → ноль. Рычаги #2–4 не понадобились. Правки промпта на 7B исчерпаны, по умолчанию — `gen-7b-v2`.

## Фаза 2 — Корпус + embed_text + улучшение поиска + упаковка (`v0.2.0`→`v0.3.0`)

**Расширение корпуса до 2021–2026 — это миграция, а не перебор вариантов.** Корпус — то, на чём построена оценка; меняя его, двигаешь саму оценку, а не проверяешь переменную против неё.

**Порядок миграции:** переиндексация → переразметить/расширить набор вопросов (несколько правильных статей для Q2, перепроверка существования для Q1) → пересъём базового прогона → новый суффикс версии. Плюс инфраструктура: DVC, S3, Docker, CI.

**Сделано (→ `v0.3.0`):** переиндекс 2021–2026 (5.3×, embed_text) ✓ · оценка RAGAS→MLflow ✓ · переразметка набора (базовый прогон `gen-7b-v4`) ✓ · **запас качества + сравнение способов поиска** (реранкер +16 п.п. за флагом; BM25 отброшен) ✓ · ограничение частоты запросов к судье **исправлено** ✓ · **чат-демо** (Gradio) ✓. **Ближняя цель — упаковка в портфолио** (витрина README, релиз `v0.3.0`). Инфраструктура (DVC/S3/Docker/CI) отложена — не приоритет под текущую цель.

**↳ Реранкер и гибрид — проверены на трудном наборе.** Они улучшают **ранжирование**, а не полноту, и на лёгком наборе (полнота на потолке) их выигрыш неизмерим. Поэтому сначала **создали запас для улучшения** (трудный набор: косвенные вопросы без терминов-подсказок, recall@1 0.40), затем замерили без судьи (`retrieval_ab.py`):
- **реранкер** (`bge-reranker-v2-m3`, статья оценивается по своему лучшему чанку): **+16 п.п. recall@1, +0.095 MRR** → ✅ принят за флагом `RERANK_ENABLED` (но ~1.1 ГБ видеопамяти рядом с 7B не помещаются в 8 ГБ → включать на GPU ≥10 ГБ);
- **гибрид (BM25 + плотный поиск, слияние RRF):** −0.07 MRR → ❌ отброшен (русские запросы / английские документы: лексика почти не пересекается);
- **фильтр по метаданным / более сильный эмбеддер** — не понадобились (полнота держится; эмбеддер — самый дорогой, в последнюю очередь).

Очерёдность «дёшево→дорого» (фильтр по метаданным → реранкер → гибрид → сильный эмбеддер) остаётся ориентиром, если всплывёт новое падение полноты.

## Фаза 3 — Агентные вопросы с несколькими источниками (`v2.0`)

**Логична только после Фазы 2.** Q1/Q2 сегодня — **одношаговые**; агент ничего не добавляет, и его выигрыш неизмерим. Мультигодовой корпус **естественно рождает вопросы с несколькими источниками**: «сравни подходы A/B/C к X», «эволюция метода Y во времени», «найди работу, объединяющую X и Y».

**В ней:** разбиение запроса на части, итеративный цикл «поиск — рассуждение», самостоятельное уточнение запроса.

**Критерий:** в наборе появился класс таких вопросов, на которых агент измеримо обгоняет одношаговый базовый прогон. Без таких вопросов — преждевременно.

## Фаза 4 — Graph / Citation RAG (исследовательское, дальше всего)

**Только как продуктовая ставка**, не ради моды. Оправдан, если делаем упор на **запросы о происхождении и цитировании**: «что выросло из статьи Y», «родословная метода X», «кто процитировал Z». arXiv для этого структурно богат (цитаты, авторы, датасеты, методы).

**Цена:** построение графа (извлечение цитат / сущностей / связей) — крупная инженерия.

**Двойной критерий:** (а) корпус даёт связную структуру (Фаза 2 сделана); (б) в оценке есть вопросы графового типа. Иначе — сложность, которую не оправдать и не измерить.

## Экономика оценки (сквозное ограничение)

Один прогон оценки ≈ **$0.04/вопрос → ~$2 за 50 вопросов** (судья Haiku) и упирается в **лимит входных токенов в минуту** (Haiku, Tier 2: 450K токенов/мин), а не в число запросов в минуту. Главный драйвер цены — `retrieval_relevance` (отдельный вызов судьи на каждый чанк).

Число параллельных запросов к судье ограничено (`MLFLOW_GENAI_EVAL_MAX_WORKERS`) под этот лимит. → Поэтому экспериментов **мало и они прицельные** (A/B по одному изменению), а не массовый перебор. Это ограничение и держит карту честной: не строим то, что не окупится **измеримым** выигрышем.

## Когда и как пересматривать эту карту

**Пересматривай, когда:**
- **данные противоречат гипотезе фазы** — напр. оказалось, что потеря не в генерации, а в поиске (тогда Фаза 0 и Фаза 2 меняются местами/содержанием);
- **меняется область или цель продукта** (другой домен, другой класс вопросов, другой пользователь);
- **критерий фазы достигнут — или оказался неверным** (метрика не двигается рычагом, на который ставили);
- **появился более дешёвый/прицельный измеримый рычаг**, которого не было на момент написания;
- **внешние условия сдвинулись** (лимиты API, доступный GPU, новые модели/эмбеддеры).

**Как пересматривать:**
- правь этот файл и коммить — история изменений карты сама по себе ценна;
- не бойся **переставлять, дробить или выкидывать** фазы;
- любое новое «надо сделать X» прогоняй через [Принцип](#принцип): где данные показывают потерю и **сможет ли оценка увидеть выигрыш**. Если измерить нельзя — сначала почини измерение (набор вопросов / метрику), потом строй функцию.
